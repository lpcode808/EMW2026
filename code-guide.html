How to learn from this code

Predict, test, observe, then explain

One of the strongest ideas in the attached teaching prompt was to treat code like a mystery you can investigate. That works beautifully here. Instead of trying to memorize everything, move through the code with a simple loop:

1

Predict

Read a variable or function name and guess what it does before you scroll further.

2

Search

Use find-in-page for names like renderSchedule or saveNote.

3

Change one thing

Edit one color, label, or message at a time so you know which change caused which result.

4

Observe

Refresh the browser, compare what changed, and then explain it back in your own words.

Tip: learners build confidence faster when they can see cause and effect. The best first edits are usually visual or text-based.

Big picture first

The mental model

The live app works like a loop: data gets turned into HTML, buttons listen for taps, and notes are saved in the browser. If you understand those four moves, most of the file stops feeling mysterious.

1

Data

The app stores conference information in JavaScript arrays like SCHEDULE and SPEAKERS.

2

Render

Functions such as renderSchedule() loop through that data and build the visible cards on screen.

3

Interact

Event listeners wait for taps on tabs, accordions, chips, and note buttons, then call the right helper function.

4

Remember

Notes are stored with localStorage, which means they stay on the same device even after refresh.

1. Structure

HTML gives the app its rooms

Think of HTML as the blueprint of the page. It creates the header, the three tab panels, the export modal, and the toast message. The JavaScript later fills some of those rooms with session cards and speaker cards.

Core layout

Shortened from the real app structure in index.html.

  <section id="scheduleTab" class="tab-content active"></section>

<div class="modal-overlay hidden" id="exportModal">...</div>

Why this matters

The HTML sets up places the JavaScript can target later with getElementById() and class names.

The header stays visible while you scroll.

Tabs switch which section has the active class.

The modal and toast already exist, so the script only needs to show or hide them.

2. Look and feel

CSS makes the site readable on a phone

The CSS in this project does more than make things pretty. It creates large tap targets, clear contrast, sticky navigation, smooth accordion motion, and a visual system that feels consistent across the whole app.

Design tokens

These custom properties act like named paint buckets the rest of the stylesheet can reuse.

  --bg: #001329;

Accordion animation

The expanding session rows work by animating max-height.

.session-body {
  max-height: 0;
  overflow: hidden;
  transition: max-height 0.42s cubic-bezier(0.16, 1, 0.3, 1);

Mobile-first spacing

Most spacing starts small and readable for phones, then expands on larger screens.

Safe tap targets

Buttons and triggers are designed around roughly 44px minimum height so they are easy to tap at a busy event.

Reusable colors

Changing one token updates many parts of the UI, which is much easier than hunting through dozens of hard-coded hex values.

3. Behavior

JavaScript turns data into a working app

The JavaScript is the engine room. It stores conference data, renders lists, wires buttons to actions, opens accordions, saves notes, and keeps the tab experience in sync.

Render the schedule

The app loops through the schedule data and builds one HTML string for all session cards.

function renderSchedule() {
  const html = SCHEDULE.map(session => `...`).join('');
  document.getElementById('scheduleList').innerHTML = html;
  attachScheduleEvents();
}

Listen for taps

After the cards exist, the script attaches event listeners to the new buttons.

root.querySelectorAll('.session-trigger').forEach(btn => {
  btn.addEventListener('click', () => toggleSession(btn.dataset.id));
});

Switch tabs

One helper function updates both the tab buttons and the visible panels.

function activateTab(target) {
  document.querySelectorAll('.tab-btn').forEach(btn => {
    const isActive = btn.dataset.tab === target;
    btn.classList.toggle('active', isActive);
  });
}

Protect the HTML

The esc() helper converts special characters before putting text into HTML strings.

function esc(str) {
  return str
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;');
}
Important idea: rendering and event wiring happen in that order. First the code puts new HTML on the page, then it teaches those new buttons what to do when tapped.

4. Saved data

Why notes stay on this device

The app does not send notes to a server. Instead, it uses the browser's local storage. That makes the app simple and private for a conference setting, but it also means the notes live only in the same browser on the same device.

Read saved notes

The app loads whatever was saved earlier under one key.

const NOTES_KEY = 'emw2026_notes';

function getNotes() {
  const raw = JSON.parse(localStorage.getItem(NOTES_KEY) || '{}');
  return raw;
}

Write saved notes

When a user saves, the app updates the note object and stores it back as JSON text.

function persistNotes(notes) {
  localStorage.setItem(NOTES_KEY, JSON.stringify(notes));
  updateNoteCount();
}

Why JSON?

localStorage only stores strings, so objects and arrays need to be converted with JSON.stringify().

Why timestamps?

Each note stores when it was created and updated so the Notes tab can show useful context back to the learner.

What is the tradeoff?

If you switch browsers or clear storage, the notes will not follow you. Simplicity and portability often trade off against each other.

What to ask yourself

Good questions while reading code

Why store conference data in JavaScript instead of writing every card by hand?

Because the cards all share the same structure. When data is in arrays, one render function can create many cards, which cuts repetition and keeps edits more consistent.

Why does the app build HTML strings instead of using a framework?

For this project, simplicity wins. A single-file app with no build step is easy to deploy on GitHub Pages, easy to open locally, and easy for learners to inspect without installing tools.

Why is there an esc() function?

Because when text is inserted into HTML, special characters can accidentally break the page or create unsafe markup. Escaping turns those characters into plain text first.

Why attach event listeners after rendering?

Buttons need to exist in the page before JavaScript can find them. The render step creates them, then the event step connects behavior to them.

What would be the next level up from here?

A next step might be separating data, styles, and behavior into different files, or moving to a framework once the project becomes too large for one main app file to stay comfortable.
